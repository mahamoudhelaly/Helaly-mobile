# HellyBuy — تطبيق متجر إلكترونيات (React Native)

تطبيق متجر إلكتروني للإلكترونيات (هواتف، لابتوبات، شاشات، سماعات، وإكسسوارات) مبني بـ React Native و Expo، بهوية بصرية داكنة مخصصة، ومتصل بالكامل بـ [Backend API](https://github.com/mahamoudhelaly/Helaly--mobile) حقيقي.

## نظرة عامة

HellyBuy تطبيق يغطي رحلة المستخدم الكاملة في متجر إلكتروني: تسجيل الدخول، تصفح الفئات والمنتجات، إضافة للسلة، إتمام الطلب، ومتابعة الحساب والطلبات السابقة — كل ذلك متصل بـ API حقيقي مبني خصيصاً لهذا التطبيق (وليس بيانات وهمية ثابتة).

## المميزات

- **مصادقة كاملة** — تسجيل دخول وتسجيل حساب جديد، مع تخزين آمن للتوكين عبر `expo-secure-store`
- **تصفح الفئات والمنتجات** — عرض شبكي (Grid) مع صور وأسعار حقيقية
- **صفحة تفاصيل منتج** — معرض صور (Carousel)، وصف، ومنتقي كمية
- **سلة تسوق** — إضافة، تعديل الكمية، حذف — مُدارة عبر React Context
- **إتمام الطلب (Checkout)** — إدخال عنوان التوصيل وبيانات بطاقة، وإنشاء طلب فعلي عبر الـ API
- **الملف الشخصي** — بيانات المستخدم، نقاط الولاء، وتسجيل الخروج
- **سجل الطلبات** — عرض كل الطلبات السابقة مع حالتها (قيد الانتظار، تم الشحن، إلخ)
- **المفضلة (Wishlist)** — عرض وحذف المنتجات المحفوظة

## التقنيات المستخدمة

| الفئة | الأداة |
|---|---|
| Framework | React Native (Expo) |
| التوجيه | Expo Router (file-based routing) |
| اللغة | TypeScript |
| إدارة الحالة | React Context (للسلة) |
| التخزين الآمن | expo-secure-store |
| الاتصال بالـ API | Fetch API مع طبقة تجريد مخصصة |

## التشغيل محلياً

### المتطلبات
- Node.js 18+
- تطبيق [Expo Go](https://expo.dev/go) على موبايل حقيقي (أو محاكي Android/iOS)
- [Backend API](https://github.com/mahamoudhelaly/Helaly--mobile) شغال (محلياً أو عبر رابط عام)

### الخطوات

```bash
# 1. تثبيت الحزم
npm install

# 2. إعداد متغيرات البيئة
cp .env.example .env
# عدّل EXPO_PUBLIC_API_URL برابط الـ Backend الفعلي

# 3. تشغيل السيرفر
npx expo start
```

امسح QR Code الناتج بتطبيق Expo Go.

> **ملاحظة:** عند التطوير على موبايل حقيقي متصل بشبكة منزلية، استخدم عنوان IP المحلي للجهاز (`192.168.x.x`) أو نفق مثل [ngrok](https://ngrok.com) إذا واجهت مشاكل في الاتصال بالشبكة المحلية.

## قرارات تصميمية

**لماذا React Context للسلة بدلاً من مكتبة إدارة حالة خارجية؟**
حجم وتعقيد حالة السلة في هذا التطبيق (قائمة عناصر + كمية + سعر إجمالي) لا يستدعي إضافة تبعية خارجية مثل Redux أو Zustand. الـ Context API كافٍ تماماً وأبسط للقراءة والصيانة.

**لماذا السلة محفوظة في الذاكرة فقط (وليس في قاعدة بيانات)؟**
في هذا الإصدار، السلة تُجمَّع في التطبيق وتُرسَل دفعة واحدة كطلب (`POST /orders`) عند الضغط على "تأكيد الطلب". هذا يبسّط التزامن (Sync) بين الجهاز والسيرفر، على عكس نموذج "سلة محفوظة في قاعدة البيانات" الذي يصبح مفيداً أكثر عند الحاجة لاستعادة السلة بين الأجهزة.

**لماذا واجهة دفع بصرية بدون بوابة دفع حقيقية مفعّلة؟**
الهدف من هذا الإصدار إثبات تكامل تجربة المستخدم الكاملة (UX flow) من السلة إلى تأكيد الطلب. الطلب الفعلي يُسجَّل في قاعدة البيانات الحقيقية عبر الـ API، بينما معالجة الدفع نفسها (Stripe/Paymob) تتطلب حساب تاجر موثّق، وهي خطوة منفصلة عن منطق الطلبات.
